## Decisions made

- **Continue affordance stays user-confirmed, not auto-advance.** Initial misread of feedback led to a brief auto-advance proposal; reverted. The brief's original journey is correct: LLM signals `readyToAdvance`, user clicks `Continue`, runtime validates checklist + advances.
- **Slash-command popover/autocomplete deferred to v1.1** — parse-on-submit only in v1. The brief's open question called this out; this session honoured the deferral.
- **`SkillLaunchButton` molecule (page-launched "Generate / Refresh" buttons) deferred** until DNA-02 introduces the first real consumer. The v1 example skill (`hello-world`) is launched via slash command only, so no need to abstract the button shape now.
- **State extraction strategy: two LLM calls per turn.** `streamText` (Gemini Pro) for prose, then a follow-up `generateObject` (Claude Haiku 4.5) in `onFinish` for structured state. Cleaner than tool-call discipline; one extra round-trip on a fast model. Documented in the brief's amendment from 2026-04-30 (state extraction edge case).
- **Skill conversation titles are `<Skill name> · <date>` at creation, not auto-generated.** Auto-titling could mislead once a skill is in flight.
- **`InlineWarningBanner` extracted now (not later)** because OUT-01b's spec already calls for the same shape ("State update failed at…"). Two consumers justify the molecule even though one hasn't built yet.
- **State-extraction prompt explicitly lists checklist IDs and the inclusivity rule.** Caught during smoke testing — without it, `gathered.topicCurious` populated but `checklist[0].filled` stayed `false`. Important for staged skills like DNA-02.
- **No new ADR this session** — all decisions are implementation-level or were already covered by ADR-008 (skills as a separate primitive).
