## Decisions made

- **Pluggable context-tab contract** — one tab visible at a time, multiple tabs registerable; tabs co-located with their data domain (skill-state lives in `lib/skills/`); registry imports tab modules. Captured in the layout spec + `chat-context-pane-template.md`.
- **`SkillSummary` server-side projection** — server pages compute `Skill` → `SkillSummary` via `toSkillSummary` and pass the projection down. The skill-state tab consumes summaries instead of importing the server registry. Keeps `node:fs` (loaded transitively via `lib/skills/load-brief`) out of the client bundle. **Pattern to follow:** any future context tab whose data lives in a server-only module should expose a client-safe summary type and a server-side projection helper, not import the source module from the tab.
- **Pane defaults to closed (rail-only).** First dev-mode pass had the panel open by default; chat felt cramped. Flipped via `DEFAULT_PANE_PREFS` + migration 0035. The rail-icon click is the single open affordance, removing the parallel "open pane" toggle that was previously in chat-area.
- **`InlineWarningBanner` tone extension.** Reversed the original "no variant prop, ship sibling molecules" rule because OUT-01b had a real second consumer (success completion banner) with the same shape. The molecule spec now codifies the rule as: extend tone only when a real consumer needs it; don't add `error`/`info` tones speculatively.
- **`RailIcon` is its own molecule.** Forks `IconButton` because the chrome contract differs (no border, surface-aware selected state, dot + adornment). 40px hit area vs IconButton's 32px because the rail is a primary nav surface.
- **Upload-route v3 defaults** (BUG-01 partial close, option A from this session). Captured inline in `app/api/sources/route.ts` and the BUG-01 backlog entry. Below the ADR bar — scoped to one route, reversible, and INP-12's inbox UI will likely revisit.

No new ADRs this session. ADR-008 (skills vs content-creation, from OUT-01a) remains the load-bearing architectural ADR for the chat stack.

**Skill update (Step H):** added a stack-convention rule to `feature-build` SKILL.md describing the server-only-registry → client-safe-summary projection pattern. The rule references `lib/chat-context-pane/skill-summary.ts` as the canonical example, and warns that Turbopack will pull `node:fs` into the client bundle if a client component imports a registry that transitively loads briefs / file-based catalogues. Discovery cost was a mid-build refactor this session; the rule is meant to surface it at planning time for future sessions.

## Context for future sessions

- **Pane state is per-conversation for the selected tab, per-brand for open/width.** Don't move pane width to per-conversation without a real reason — current pattern is intentional.
- **Skill-state tab status is always `'active'`.** Never returns `'empty'` or `'hidden'`. The skill picker IS the empty state; that's why the rail icon is always reachable. New tabs may want different status semantics.
- **`node:fs` sneaks in via `lib/skills/load-brief.ts`.** Anything that imports `lib/skills/registry` must stay server-side. Use `toSkillSummary` to project to a client-safe shape. Pattern likely repeats for other server-only registries (content-types, prompt fragments).
- **`PaneHighlightPulse` doesn't pulse on first mount.** It captures `pulseKey` on first render and only pulses on subsequent changes. If a future consumer wants an initial-mount pulse, the molecule needs a `pulseOnMount?: boolean` flag.
- **Rail width is 56px, hit area 40px.** Future rail-icon adornments (e.g. red error dot, info dot) need to fit in the same 40px square without overlapping the existing dot (top-right) or check adornment (bottom-right). Two more corner slots are available.
- **Migrations 0031 and 0035 are OUT-01b's; 0032/0033/0034/0036 are INP-12's.** Both sets are applied to the live DB. The `_journal.json` file is shared bookkeeping; whichever session commits first commits the current state.
- **Brand-id is hardcoded** in `app/actions/brand-preferences.ts` matching the existing pattern. When auth-driven brand resolution lands (post-INF-05), all server actions need to swap from the hardcoded constant to a session-based lookup — this one included.
- **Upload-route v3 defaults are interim.** When INP-12's inbox UI lands, the upload flow may want to ask for `sourceType` + `authority` at upload time, or auto-classify based on more context. Current behaviour (catch-all + flag for inbox triage) keeps things working but isn't the long-term shape.
