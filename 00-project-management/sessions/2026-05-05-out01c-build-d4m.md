## What was done

### Brief (OUT-01c)
File: `01-design/briefs/OUT-01c-llm-database-write-tool.md` (new, status `approved`).

Locked the three-property contract: schema-aware (LLM tools generated from Drizzle + writes-lib `WRITABLE_FIELDS`), confirmation-by-default (UI diff card in OUT-01b pane, never in-prose hand-waving), auditable (single `entity_writes` table covering UI + LLM + skill writes).

Five use cases captured: skill terminal save (UC-1), ad-hoc chat write (UC-2), in-skill mid-conversation write (UC-3), create-from-data (UC-4), create-via-generation (UC-5). v1 scope: all DNA singulars + plurals + ideas (update + create where applicable); offers excluded from direct-create surface because the DNA-04 multi-stage flow is skill-shaped. Postgres only — graph writes split out to OUT-01c-graph.

Architectural decisions worked through in the brief:
- `onComplete` skill saves bypass `db_update_bot` and call `lib/db/writes/*` directly. Skill flow is the user's authorisation; an LLM hop on a deterministic save adds latency, cost, and a failure mode for nothing.
- Single sub-agent `db_update_bot` (not split per entity-class). Tool decision tree stays one level deep; future graph writes land here as additional tools without restructuring.
- Audit log lives in one `entity_writes` table covering UI-driven and LLM-driven writes. UI server actions retrofit to log via the writes lib.
- `pending_writes` is its own table (not just `context_pane_state` jsonb). Survives reload; decided rows preserved for trace continuity.
- v1 ship gate is three end-to-end exercises: `update_dna_brand_meaning`, `create_idea`, `request_dna_audience_segment_generation` — covering the three op classes.

### Build (OUT-01c)

**Phase 1 — DB schema** (migration `0037_nifty_roughhouse.sql`)
- New table `entity_writes` — append-only audit log. `actor` shape `'ui:<path>'` / `'chat:<conversationId>'` / `'skill:<id>:conversation:<id>'`. Index `(brand_id, entity_type, entity_id, at)`.
- New table `pending_writes` — LLM-proposed writes awaiting confirmation. Lifecycle `pending` → `confirmed` | `rejected`; decided rows persist. Index `(conversation_id, status)`.
- `messages.metadata` jsonb column for system-message kind discriminators. `messages.role` kept as `varchar(20)` (no enum — `'system'` is a code-level convention).

**Phase 2 — Writes lib** (`02-app/lib/db/writes/`)
- 9 entity write modules (`dna/business-overview`, `dna/brand-meaning`, `dna/value-proposition`, `dna/audience-segment`, `dna/offer`, `dna/platform`, `dna/tone-of-voice`, `dna/knowledge-asset`, `ideas`).
- Each exposes `WRITABLE_FIELDS` + `getCurrentValue` (for diff) + `updateField`. Plurals add `create` (where applicable); audience-segment also exposes `kickoffGeneration` (sync wrapper around the existing GEN-01 `lib/generation/audience-segment.ts`).
- `audit.ts` — `logEntityWrite` helper. Best-effort logging — the Neon HTTP driver does not support transactions, so the entity write and the audit row write happen serially with structured error logging on audit failure. The brief had specified atomic transactions; for a single-user app with a simple insert-only audit table, the practical risk is negligible. Decision recorded in OUT-01c brief decisions log.
- `entity-breadcrumb.ts` — pure helpers for human-readable entity names + write summaries (used by `PendingWriteCard` + the in-chat divider text).
- `index.ts` — `WRITES_BY_ENTITY` registry + `getEntityWrites(entityType)` lookup.

**Phase 3 — Pending-writes plumbing**
- Query module `02-app/lib/db/queries/pending-writes.ts` — list/get/insert/markStatus.
- Server actions `02-app/app/actions/pending-writes.ts` — `confirmPendingWriteAction` and `rejectPendingWriteAction`. Confirm executes the entity write via writes-lib, marks the pending row `confirmed`, injects an in-chat `'system'` message with `metadata.kind = 'write-confirmation'`, revalidates returned paths. Reject just marks status; no DB write, no in-chat message.
- `addMessage` extended to accept `'system'` role + `metadata` opts field.

**Phase 4 — `db_update_bot` sub-agent** (`02-app/lib/skills/sub-agents/db_update_bot/`)

## Decisions made during build

- **Audit logging is best-effort, not transactional.** Neon HTTP driver doesn't support transactions. Acceptable for a single-user app with simple inserts on the audit table. Decision recorded in brief decisions log.
- **Server-action retrofit is partial.** Plan was 7 files; delivered cleanly on 3. Other 4 deferred as a side-quest. Worth a follow-up commit before the audit log can be considered the source of truth.
- **Async generation (UC-5) is sync-wait inside the server action.** Per Refinement A confirmed at plan time. The pending-write card stays in `saving` state for ~30s while `confirmPendingWriteAction` awaits `kickoffGeneration`. Honest UX (visible loading state) but blocks parallel generations during the wait. Vercel function lifetime is well within budget.
- **`pending-writes` tab priority = 20** (higher number = lower in rail). Skill-state at 10 stays first; pending-writes shows below it when both are active.
- **`messages.metadata` is the system-message kind discriminator**, not a separate `messages.kind` column. `metadata.kind = 'write-confirmation' | 'generation-complete' | 'generation-failed'` with op/entity_type for tooltips.
- **`pending-writes-tab.tsx` state machine uses `useState` + `removedIds` set, not a ref.** Initial implementation read a ref during render; Next.js refused. Switched to a single `Record<id, state>` keyed by pending-id with `?? 'idle'` fallback for unknown keys.

## Bugs / surprises caught during build

- **Stash-pop hazard.** Mid-build I ran `git stash` to confirm a typecheck error was pre-existing, then `git stash pop` lost my modifications to existing files (kept new files but reverted edits). Fix was to `git checkout stash@{0} -- <paths>` to restore explicitly. **Lesson:** when verifying pre-existing state mid-build, save changes via commit not stash, or run the verification in a separate worktree.
- **`load-brief` uses `node:fs`.** The `db_update_bot` brief loader runs server-only — same constraint as `retrieval_bot`. The whole sub-agent module never imports into a client component. Confirmed via the chat-area client/server boundary: tab registry stays client-only and never imports the registry that pulls in briefs.
- **Tools record type variance.** AI SDK `tool<I, O>()` produces strongly-typed values that don't fit a homogeneous record key. Widened to `Record<string, any>` with eslint-disable, matching the pattern `createRetrievalTools` already uses.
- **`scripts/generate-db-update-bot-schema.ts` needed `dotenv` wrap.** The generator transitively imports `lib/db/index.ts` which calls `neon(process.env.DATABASE_URL!)` at module load. Fixed with `dotenv -e .env.local --` prefix on the npm script — matches existing `db:generate` / `db:migrate` scripts.
